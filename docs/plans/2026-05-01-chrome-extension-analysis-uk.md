# Аналіз: Chrome Extension як альтернативна архітектура

**Дата:** 2026-05-01
**Контекст:** Розглядається альтернатива поточному плану v0.1 (iced Rust-native desktop app). Користувач запитав, що якщо побудувати все на Chrome Extension.

---

Чесний аналіз: Chrome Extension спростить три речі і зламає три речі. Чи буде це чистим виграшем — залежить від того, які частини специфікації для тебе критичні.

## Що добре працює в Chrome Extension

| Розділ специфікації | Як |
|---|---|
| **Захоплення екрана** | `chrome.desktopCapture.chooseDesktopMedia()` → `MediaStream` для вкладок, вікон, повного екрана — абстракція платформ безкоштовна |
| **Вебкамера + мікрофон** | `getUserMedia()` — той самий потік дозволів браузера, без `nokhwa`, без `Info.plist` |
| **Дистрибуція** | Chrome Web Store, встановлення в один клік, автоматичні оновлення — повністю прибирає роботу з §9 (нотаризація + cargo-dist), ~3 дні економії |
| **Кодування** | API `MediaRecorder` нативно видає WebM або MP4 (H.264) — не треба бандлити ffmpeg sidecar (~30 МБ економії) |
| **Зберігання** | File System Access API (користувач один раз обирає `~/JarvisStudio`, браузер запам'ятовує дескриптор) |
| **Шеринг** | `fetch()` до RustFS працює як є — дизайн §8 не змінюється |
| **Vision LLM** | Прямий HTTP до Anthropic / OpenAI / OpenRouter / Ollama — дизайн §6 не змінюється |

## Що ламається

| Розділ специфікації | Чому |
|---|---|
| **§4 архів H.265** | `MediaRecorder` не видає HEVC надійно. Доведеться перейти на H.264 / VP9 / WebM — втрачаєш обґрунтування "~50% менші файли ніж H.264" |
| **Системне аудіо macOS (§4)** | У Chromium давній пробіл: `getDisplayMedia({ audio: true })` НЕ захоплює системне аудіо на macOS. Користувачам знадобиться BlackHole/Loopback — саме той kext, який ми вирішили не постачати |
| **§7 плагінні агентські бекенди** | Chrome Extension не може спавнити CLI `claude` / `codex`. Єдиний вихід — `nativeMessaging` хост, тобто нативний бінарник, який треба інсталювати і підписувати. Ти знову повертаєш всю роботу, яку extension мав ліквідувати |
| **§4 фрагментований MP4 з заданим темпом ключових кадрів** | `MediaRecorder` видає те, що видає — ти не можеш фіксувати `-g 150 -force_key_frames`. Гарантія "≤5с втрати при крашi" перетворюється на "як вирішить Chrome" |

## Три життєздатні форми

### A. Чистий extension (без нативного хоста)

Викидаєш агентів §7, викидаєш H.265 §4, погоджуєшся на відсутність системного аудіо на macOS. Тепер ти будуєш браузерний клон Loom. Швидше випустити; помітно менш амбітно ніж поточний дизайн.

### B. Extension + маленький native messaging хост

Extension робить захоплення/UI, бінарник-хост робить ffmpeg-постобробку + субпроцеси агентів. **Тут стає цікаво:** отримуєш захоплення без інсталяції, але все одно постачаєш нативний бінарник для агентів. Узгодженість із v0.1 планом висока; iced UI замінюється на HTML+JS extension, але крейти `recorder-core` / agents / sharing виживають.

### C. Web app (PWA) замість extension

Те саме що A, але без залежності від Chrome Web Store. File System Access API — шар зберігання. Втрачає API доступні тільки extension (`tabCapture`, привілейовані варіанти `desktopCapture`).

## Моя рекомендація

Залишайся з планом на iced для v0.1 — три зміни стеку за три дні (Tauri → iced → Chrome Ext) — це ризик, а не прогрес.

**Але** якщо реальний драйвер цих пивотів — "взагалі без інсталятора", правильна відповідь, мабуть, **форма B** як канал дистрибуції для v2:

- Випускай iced як v1.0 для досвідчених користувачів
- Додавай Chrome Extension як фронтенд, що говорить із тим самим Rust-ядром через native messaging для звичайних користувачів

Це покриває обидві аудиторії без розгалуження кодової бази.

## Питання, на які треба відповісти перед рішенням

Яка справжня мотивація?

- Якщо це **фрикція встановлення** → форма B (extension + native messaging хост на v2)
- Якщо це **"не хочу писати Rust UI"** → це той самий пивот на iced, який ти вже зробив; extension був би іншим типом UI-роботи (TS/HTML), не меншим
- Якщо це **"хочу веб-деплоймент"** → це реальне рішення про позиціювання продукту, а не просто заміна стеку

## Що це означає для поточного v0.1 плану

Якщо ти зараз вирішуєш повністю пивотити на Chrome Extension:

| Файл | Стан |
|---|---|
| `docs/plans/2026-05-01-jarvis-studio-design.md` | Доведеться переписати §3, §4, §5, §6, §7, §9, §11 — всі рішення про захоплення, кодування, агентів, дистрибуцію стають іншими |
| `docs/plans/2026-05-01-vibe-tests.md` | Більшість gaps стають N/A, треба перепрогнати vibe-tests з новими персонами для browser context |
| `.omc/plans/2026-05-01-v0.1-recording-pipeline.md` | План повністю переписується — Cargo workspace замінюється на TypeScript extension з manifest v3 |
| `CLAUDE.md` | Інваріанти про iced, nokhwa, ffmpeg sidecar, atomic state writes, SealedExport — більшість стає неактуальною |

Якщо обираєш форму B (extension + native host) — це менший пивот: rust core / agents / sharing крейти зберігаються, треба тільки додати extension як другий фронтенд.

Якщо обираєш форму A (pure extension) — це повністю інший продукт. Тоді чесніше відмаркувати поточний дизайн як "v1 design (deprecated)" і почати новий design doc з нуля.
